These terms are commonly used in application/cloud migration strategies (often called the “6 R’s” or “5 R’s” of modernization).
Here’s a clear and simple explanation of each:

1. Rehost (Lift and Shift)
What it means: Move an application as-is from one environment to another (e.g., on‑premises → cloud) without changing code.
Focus: Speed and simplicity.
Example: Moving a VM from your data center to AWS/Azure VM.
✅ Pros:
Quick migration
Low risk
❌ Cons:
Doesn’t use cloud-native benefits
May not reduce costs much

2. Replatform (Lift, Tinker, and Shift)
What it means: Move the application to a new platform with minor optimizations, but no major code changes.
Focus: Improve performance using managed services.
Example:
Moving from self-managed database → managed cloud DB (Azure SQL, RDS)
Moving app to containers without redesigning logic
✅ Pros:
Better performance than rehost
Some cloud benefits
❌ Cons:
Limited modernization

3. Refactor (Re-architect)
What it means: Redesign and rewrite parts (or all) of the application to fully use cloud-native features.
Focus: Scalability, resilience, and innovation.
Example:
Monolith → microservices
Adding serverless components (Azure Functions, Lambda)
✅ Pros:
Best performance and scalability
Fully cloud-optimized
❌ Cons:
Time-consuming
Higher cost and complexity

4. Redeploy
What it means: Deploy (or deploy again) an application to an environment, often as part of CI/CD pipelines or migration.
Not strictly part of “R’s” but commonly used in DevOps.
Example:
Deploying a new version of your app to Kubernetes
Redeploying after configuration changes
✅ Use cases:
Updates
Fixes
Environment changes

5. Repurchase:-
Replace the existing application with a different product or a SaaS version of the same product instead of migrating the current application to AWS.
Why Repurchase?
Eliminates infrastructure management.
Reduces maintenance and upgrade effort.
Vendor handles scalability, patching, and availability.
Often aligns with cloud-first strategies.

IIS(Internet Information services) Config:-
1. It is a service installed in the server, then this server becomes web server.
2. A web server is a system that stores websites and delivers web pages to users over the internet.
3. IIS is a web server software created by Microsoft that is used to host and manage websites and web applications on Windows servers. It is similar to Apache, nginx web servers.
4. IIS is used for both onpremise windows enviornment and also Cloud services like Azure VM and AWS EC2.
5. Today many apps:
Don’t use IIS at all
Run on:
Linux servers
Containers (Docker/Kubernetes)
Cloud platforms (Azure App Service, etc.)

✅ IIS – One-line points

HTTP/HTTPS handling → Receives browser requests and sends responses.
Security → Provides built-in SSL, authentication, and request filtering.
Reverse proxy & load handling → Manages incoming traffic efficiently before reaching your app.
Process management → Auto-restarts apps and isolates them using Application Pools.
Multiple app hosting → Hosts multiple apps on one server with clean URLs.
Logging & monitoring → Tracks requests, errors, and performance for debugging.
Enterprise integration → Works with Windows authentication and Active Directory.

IIS SITES:-
IIS Sites are simply the websites or web applications configured inside IIS.
Simple Example
Suppose you have a Windows server running IIS. Inside IIS, you can create multiple sites:
IIS Server
├── Site1 → app.company.com
├── Site2 → api.company.com
└── Site3 → intranet.company.com
Each of these is called an IIS Site.

What does an IIS Site contain?
An IIS Site typically has:
Website files (HTML, ASP.NET, APIs, etc.)
Bindings
  Domain name
  Port number
  Protocol (HTTP/HTTPS)
Application Pool association

Example:
Site Name: EmployeePortal
Domain: employee.company.com
Port: 443
Protocol: HTTPS
Physical Path: C:\Apps\EmployeePortal
